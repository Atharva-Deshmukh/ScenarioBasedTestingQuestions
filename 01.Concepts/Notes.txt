                                        What is Software Testing?
                                        -------------------------

Testing is a set of activities to:
- Discover defects
- Evaluate quality
- Reduce risk

- Testing is NOT just execution as it includes planning, analysis, design, 
  implementation, monitoring, control, and reporting.

-----------------------------------------------------------------------------------------------------------------

                            Why Do We Say "Testing Is Context Dependent"?
                            --------------------------------------------

Because the Testing approach depends/adapts based on the type of product, domain, risks, users.
There is NO one-size-fits-all testing strategy.

- Banking/Healthcare -> Focus on High security, compliance, accuracy required.
- Gaming Apps -> Focus on performance.
- E-commerce -> Focus on usability and user experience.

Different Project Types Need Different Testing
- Startup MVP -> Fast testing, exploratory, rapid feedback.
- Enterprise System -> Detailed documentation, regression cycles.
- Safety-Critical System -> Rigorous validation and certification testing.

User Expectations Differ
- Internal tool -> Basic usability may be acceptable.
- Customer-facing product -> High usability and reliability required.

Technology Stack Matters
- Web app -> Browser compatibility testing.
- API -> Contract and integration testing focus.

-----------------------------------------------------------------------------------------------------------------

                            Importance of Testing
                            ---------------------

Testers represent real-users during development and ensures that the users' needs 
are considered during development.

Testing Saves money.
    When testers are involved early in SDLC in agile processes, inputs from the test team helps identify
    gaps in requirements and designs well in advance.
    Because of these, those defects don't go to production
    Cost of fixing these defects early is far less.

Testing is necessary because defects can cause:
- Financial loss
- Reputation damage and loss of customer trust
- Injury or death (safety-critical systems)
- Legal penalties
  Testing mitigates these risks before release.

Software testing also ensures security and performance, since its not just manual testing, it involves
load testing also

                                            Verification vs validation
                                            --------------------------

Verification                           | Validation
---------------------------------------|----------------------------------------
Are we building the product right?     | Are we building the right product?

Ensures work products conform to       | Ensures the final product meets user and stakeholder needs
specified requirements                 | 

Often analytical and review-based      | Typically evaluation- and execution-based

Checks against requirements            | Checks against user needs


WHEN DOES IT HAPPEN?
    Verification: Throughout SDLC phases (requirements, design, coding).
    Validation: After implementation, during system testing and user evaluation.


WHO IS INVOLVED?
----------------

    Verification:
    - Business Analysts/Product Owners (requirement reviews)
    - Architects (design validation)
    - Developers (code reviews, unit testing)
    - QA Team (review documents & test artifacts)

    Validation:
    - QA Team (system testing)
    - Product Owner
    - Clients / Stakeholders / Product Manager (PM Acceptance)
    - End Users (UAT, Beta testing)


METHODS USED:

    Verification Methods:
    - Requirement Reviews
    - Design Reviews
    - Code Reviews
    - Walkthroughs
    - Static Testing
    - Unit Testing
    - Integration Testing

    Validation Methods:
    - Functional Testing
    - System Testing
    - User Acceptance Testing (UAT)
    - Alpha Testing
    - Beta Testing


Example (FinTech App):

Requirement: "User can transfer up to 1,00,000 per day."
- Verification: Code enforces the 1,00,000 daily transfer limit correctly.
- Validation: Confirm whether 1,00,000/day matches real customer expectations and business needs.

                        What is Localization and Internationalization Testing?
                        ------------------------------------------------------

Internationalization
--------------------

- Ensures the application is designed to support multiple languages, regions, and cultural settings 
  without major code changes.
- Its preparing the application for multiple languages/regions.
- Its mainly Developer side
    The app is designed to:
    - Store text in translation files
    - Handle different currencies/date formats dynamically

Localization
------------

- Verifies that the application correctly adapts to a specific language, region, or culture.
- Adapting the application for a specific language/region.
- Its testing/User side
    Now the app is switched to:
    - German language
    - Euro currency
    - German date format (DD.MM.YYYY)

Internationalization = Build the capability
Localization         = Use this capability for a locale

                                        SDLC (Software Development Life Cycle)
                                        ---------------------------------------

Definition: Process followed to develop software.

Phases:
1. Requirement Gathering
2. System Design
3. Development (Coding)
4. Testing
5. Deployment
6. Maintenance

Goal: Deliver high-quality software that meets business requirements.


                                        STLC (Software Testing Life Cycle)
                                        -----------------------------------
                        
Definition: Process followed to test software.

Phases:
1. Requirement Analysis
2. Test Planning
3. Test Case Design
4. Test Environment Setup
5. Test Execution
6. Test Closure

Goal: Ensure the software is defect-free and meets user expectations.

Key Difference:
    SDLC focuses on building the product.
    STLC focuses on validating and verifying the product.


-----------------------------------------------------------------------------------------------------------------
                               
                                Difference Between Bug, Defect, and Error?
                                ------------------------------------------

- Error: A mistake made by the developer during coding.
         ex: using wrong formula to calculate discount.

- Defect: A deviation from expected behavior found during testing.
         ex: wrong discunt output in the UI.

- Bug: A confirmed defect that is logged and tracked for fixing.
       ex: the above wrong discount defect is logged on JIRA as its confirmed.

-----------------------------------------------------------------------------------------------------------------

                        What is the Difference Between Severity and Priority?
                        -----------------------------------------------------

- Severity: Measures how serious the bug is for the system.
- Priority: Defines how soon the bug should be fixed.

-----------------------------------------------------------------------------------------------------------------

                                        What is a Test Plan?
                                        --------------------

    Its a detailed document outlining the testing scope, objectives, resources,
    schedule, and activities. 

    In a Project, the test plan lists all modules to test and who will test each.

-----------------------------------------------------------------------------------------------------------------
                                       
                                        What is a Test Strategy?
                                        -----------------------

Its a high-level document that outlines the overall testing approach for the
project, including testing types, scope, tools, and responsibilities.


Test Strategy vs Test Plan
--------------------------
Test strategy is a high-level document while a test plan is a detailed document.

-----------------------------------------------------------------------------------------------------------------

                                    What is a Test Scenario?
                                    ------------------------


Its a one-line high-level situation to test. 
It represents what to test, not how.
Example:"Verify that a registered user can change their password" is a test scenario.

-----------------------------------------------------------------------------------------------------------------

                                What is a Test Case Format?
                                ---------------------------


A standard test case includes:

. Test Case ID
· Description
. Pre-conditions
. Test Steps
· Test Data
· Expected Result
· Actual Result

-----------------------------------------------------------------------------------------------------------------

                                    What is a Test Environment?
                                    ---------------------------

Its a setup of hardware, software, network, and data configured to run tests. 
It closely mimics the production environment.

Ex: a mock payment gateway

-----------------------------------------------------------------------------------------------------------------

                                        What is Test Data?
                                        ------------------

Inputs used while executing test cases, such as usernames, passwords,
Good test data covers both valid and invalid scenarios.

-----------------------------------------------------------------------------------------------------------------

                                            What is a Test Log?
                                            -------------------

Its a detailed document that records the sequence of events during test execution. 
It captures information such as the tests that were run, their execution time, who
executed them, the environment details, and any incidents or anomalies observed. 

Its helpful for debugging and tracking the progress of testing.

-----------------------------------------------------------------------------------------------------------------

                            What is Test Closure/Test status report?
                            ----------------------------------------

Wrapping up the test phase by analyzing test results, preparing test
summary reports, documenting lessons learned, and archiving test artifacts.

-----------------------------------------------------------------------------------------------------------------

                        What is an Entry and Exit Criterion in Testing?
                        -----------------------------------------------

Entry criteria --> defines when testing should begin (e.g., code freeze, stable build given).
Exit criteria  -->  define when testing is considered complete (e.g., no critical defects, 100% passing rate).

-----------------------------------------------------------------------------------------------------------------

                                    What is a Traceability Matrix?
                                    ------------------------------

Its a document that maps requirements to test cases. 
It ensures every requirement is tested and nothing is missed.

-----------------------------------------------------------------------------------------------------------------

                                    What is Acceptance Criteria?
                                    ----------------------------

These are the conditions defined before development that must be met for a feature to be accepted 
as complete by the client or product owner.


                                Acceptance Criteria Vs Requirement
                                ----------------------------------

Requirement         = Business need or functionality
Acceptance Criteria = Conditions to verify the requirement is done and acceptable

Ex:

    Requirement
    - User should be able to log in using email and password.

    Acceptance Criteria
    - With valid credentials
    - Error message appears for invalid inputs
    - Password field should be masked
    - Login should redirect to dashboard after success

-----------------------------------------------------------------------------------------------------------------

                                    What is Boundary Value Testing?
                                    -------------------------------

It involves testing values at the edge of input ranges. It's useful to catch common
boundary-related bugs.

Example: For a coupon code that works for orders above ₹500, test with ₹499, ₹500, and ₹501.

-----------------------------------------------------------------------------------------------------------------

                                    What is the Defect Life Cycle?
                                    ------------------------------


Its the journey of a bug from discovery to closure. 
States of a defect: New -> Assigned -> Open -> Fixed -> Retest -> Closed / Reopened

Example: A tester finds an issue with payment failure, logs it in JIRA, it's fixed by the dev,
retested, and closed.

-----------------------------------------------------------------------------------------------------------------

                                            What is Defect Leakage?
                                            -----------------------

It happens when a bug that should have been caught during testing is discovered by end users after deployment.

----------------------------------------------------------------------------------------------------------------

                                            Causes of defects
                                            -----------------

A defect is introduced when there is a gap between expected and actual behavior.
Defects can occur at any stage of SDLC.

Major Causes of Defects:

    Requirement Issues
    - Incomplete requirements
    - Ambiguous requirements
    - Frequently changing requirements

    Design Issues
    - Missing edge case handling

    Coding Errors
    - Logical mistakes
    - Hardcoded values

    Communication Gaps
    - Incorrect understanding of client expectations

    Time Pressure
    - Skipping reviews or testing
    - Inadequate regression testing

    Environment Issues
    - Configuration mismatch
    - Version conflicts
    - Database or server setup errors

    Third-Party Dependencies
    - API failures
    - Library/version incompatibility
    - External system downtime

                                    Levels of Software Testing
                                    --------------------------


                            Acceptance Testing  --> Top (Least in number, business validation)
                                |
                                |
                            System Testing      --> Even fewer, broader coverage
                                |
                                |
                            Integration Testing --> Fewer than unit tests
                                |
                                |
                            Unit Testing        --> Base (Maximum tests, fastest, cheapest)

                            Testing Pyramid (Levels of Testing) 

Cost and execution time increase as we move up.

Level-1: Unit Testing (Unit = Individual component, the smallest testable part of the software)
--------------------------------------------
- Tests individual components or functions.
- Performed by Developers.
- Focus: Logic correctness.
- Stubs and drivers are used to replace missing components

Level-2: Integration Testing (Module interaction)
-------------------------------------------------
- Tests interaction between integrated modules.
- Ensures modules work together correctly.
- Performed by Developers or QA.

Level-3: System Testing (Complete System)
-----------------------------------------
- Tests the complete, integrated system.
- Performed by QA team.
- Focus: End-to-end functionality against requirements.
- Includes functional and non-functional testing.

Example:
Testing full user flow from login -> transfer -> confirmation.

Level-4: Acceptance Testing (Business Validation)
--------------------------------------------------
- Final level of testing.
- Performed by Client / End Users / Product Owner / Product Manager.
- Focus: Business requirements and user needs.
- Includes UAT (User Acceptance Testing).

Example:
Client verifies that daily transfer limit meets business policy.

                                    TYPES OF SOFTWARE TESTING
                                    -------------------------

├── BASED ON ACTION
|
│   ├── Manual Testing    -> Without using any automation tools, just manually executing the workflows
│   └── Automated Testing -> Perfomed using automation tools like cypress and playwright


├── BASED ON TECHNIQUES
|
    ├── Blackbox testing
           - Tester does NOT know internal code implementation.
           - Focus in on Input vs Output behavior.
           - Testing techiques -> Equivalence partitioning, Boundary value analysis, Decision tables

│   ├── Whitebox Testing
           - Tester knows internal implementation.
           - Focus is on Code conditions and code branches coverage.
           Ex:  Code condition --> if(user.isPremium() && amount > 1000)

            Then, we need coverage for:
                Premium + >1000
                Premium + <1000
                Non-premium

│   └── Experienced Based Testing
           - Uses tester intuition and past experience.
           - It includes Error guessing and Exploratory testing
           - Experienced testers know:
                Payments fail during timeout
                Session expires unexpectedly
                Date handling breaks near month-end


Techiques for blackbox testing:

Equivalence Partitioning (EP):
- input data is divided into valid and invalid groups (partitions).
  Instead of testing every value, you test one representative value from each group.

    Ex: 
        if a feature accepts age fields from 18 to 60

        Partitions:
            Invalid: < 18
            Valid: 18–60
            Invalid: > 60
        Test Cases:
            15 → Invalid partition
            30 → Valid partition
            70 → Invalid partition

    Instead of testing every age from 1–100, only representative values are tested.

    Where Used
        Form validations
        Input fields
        Range-based logic

Boundary Value Analysis (BVA):
- focused on testing values at the edges (boundaries) because defects commonly occur there.
  
  Ex: Feature condition -> Password length must be 8 to 16 characters

    Important Boundary Values:
        7 → Just below minimum
        8 → Minimum valid
        9 → Just above minimum
        15 → Just below maximum
        16 → Maximum valid
        17 → Just above maximum

    Why Important?
        Developers often make mistakes like:
        > instead of >=
        < instead of <=

        BVA catches these errors quickly.


Decision Tables:
- Used when output depends on multiple conditions and combinations.
  In current organisation, there was search filter combination feature with lots of combinations

  Ex: 
  Login System Rules:

    Condition	        Meaning
    --------------------------
    Username valid?	    Yes/No
    Password valid?	    Yes/No

  Decision Table (out of various combinations possible):

    Username	Password	Result
    -------------------------------------
    Valid	     Valid	    Login Success
    Valid	     Invalid	Error Message
    Invalid	     Valid	    Error Message
    Invalid	     Invalid	Error Message

Static Testing:
- Testing without executing code.
- Includes: Requirement reviews, Design reviews, Code reviews, SonarQube, ESLint.
- Ex: SonarQube identified some vulnerability before deployment

Dynamic Testing: 
- Testing by executing the application.
- It includes Runtime validations, Assertions and Data-driven testing

Functional Testing:
- verifies what the system does, according to:
    Business requirements
    Acceptance criteria
- “Is the functionality working correctly?”

Non-Functional Testing: 
- verifies how well the system works.
- Focus is on Performance, Scalability, and Security
- “How efficiently and reliably does the system work?”

Non-Functional Testing - Security Testing:
- Focus: Authentication, Authorization, Data protection, API security.
- What to validate: SQL Injection prevention, JWT/session security, Password encryption, Role-based access, Sensitive data masking

Non-Functional Testing - Usability Testing
- verifies how easy and user-friendly the application is.
- Focus: UX, Navigation, Accessibility, Ease of use
- Ensure users can use the application without confusion.

Non-Functional Testing - Compatibility Testing
- verifies whether the application works correctly across:
    Devices
    Browsers
    Operating systems
    Screen sizes
    Networks

Non-Functional Testing - Reliability Testing
- verifies whether the system works consistently over time without failure.
- Focus is on stability, recovery and continuous operation


Non-Functional Testing - Maintainability Testing
- verifies how easily the application can be:
    Updated
    Debugged
    Enhanced
    Monitored

- Focus:
    Modularity
    Testability


Integration Testing - Component Integration Testing (CIT):

    - Tests integration between internal components/modules within the same application or microservice.
    - mostly Mostly mocked dependencies are used
    - Its Faster & controlled
    - Focus is on
        API/module interaction
        DB validation
        Failure handling
        Mocking dependencies


Integration Testing - System Integration Testing (SIT):

    - Tests integration between different systems/services/external applications.
    - Its Cross-system validation
    - No mock dependencies, Real/staging integrations
    - Focus is on 
        event testing
        Contract testing
        Third-party integrations
        Distributed system failures


The above were - "What kind of integration are we testing?".
There are also the approaches of integration testing.

So, overall picture:
                                                Integration Testing
                                                ├── Types
                                                │   ├── Component Integration Testing
                                                │   └── System Integration Testing
                                                │
                                                └── Approaches
                                                    ├── Big Bang
                                                    └── Incremental
                                                        ├── Top-Down
                                                        ├── Bottom-Up
                                                        └── Functional Incremental/Hybrid

    Integration Testing Approaches

    ├── Big Bang Integration
        - All modules integrated together at once and tested.
        - Disadvantage: Difficult to trace failure due to integration
        - Advantage: everything is finished before testing starts, so no stubs and divers required

    └── Incremental Integration -> Modules integrated and tested step by step. More practical and widely used
                                -> Its time consuming as drivers and stubs have to be kept ready
        ├── Top-Down
            - Testing starts from higher-level modules first.
            - Uses stubs for lower modules.
            - Ex: Frontend tested before backend is fully ready.

        ├── Bottom-Up
            - Testing starts from lower-level modules first.
            - Uses drivers for upper modules.
            - Database/API layer tested before UI.

        └── Functional Incremental / Hybrid
            - Modules integrated based on business workflow/functionality.
            - Very common in Agile + microservices.
            - Checkout flow integrated gradually ==> Cart → Payment → Order → Notification


System Testing - Feature Testing
- Testing a specific feature independently.
- Ensure the feature behaves correctly for all scenarios.


System Testing - Smoke Testing
- Basic testing to check whether the build's stability.
  Example After deployment: App launches and main page loads
- If smoke fails -> build is rejected

System Testing - Sanity Testing
- Goal is to Verify specific fix/change.
- If sanity fails -> fix is rejected

System Testing - Regression Testing
- Ensure existing features still work

System Testing - Monkey Testing
- Random testing with unexpected or random inputs/actions.

System Testing - Gorilla Testing
- Repeatedly testing one specific module deeply and aggressively.
- Goal is to Break the module and test robustness.

System Testing - Exploratory Testing
- its planned, time-boxed, and focuses on learning the app as you test.
- YOU DON'T HAVE APPLICATION KNOWLEDGE

System Testing - Adhoc Testing
- Informal testing without documentation or planned test cases but YOU HAVE KNOWLEDGE OF THE APPLICATION
- Tester casually checks: App behavior after network disconnect and Back button handling
- Its Random + informal testing


Integration testing vs System testing:

| Integration Testing                                       | System Testing                                  |
|-----------------------------------------------------------|-------------------------------------------------|
| Focuses on interfaces, communication, and data flow       | Focuses on overall system behavior              |
| Tests integrated units/components                         | Tests the entire system                         |
| Usually done after unit testing                           | Done after integration testing                  |
| Finds issues in API calls, DB interactions, message flow  | Finds functional/business flow issues           |
| Scope is limited to interacting modules                   | Scope is the whole application                  |


Acceptance Testing:
- verifies whether the application is ready for:
    Business use
    Customer usage
    Production release
- Usually performed by: Clients, POs and PMs

Acceptance Testing - Alpha Testing:
- performed internally before releasing the product to external users.
- Usually done by Internal QA team and Product team on controlled testing/staging env
- Goal is to Find major bugs before customers use the product.

Acceptance Testing - Beta Testing:
- performed by real users in a real environment before final production rollout.
- Usually done by Selected customers and external users in real world conditions
- Goal is to Collect real user feedback and discover issues missed internally.

Acceptance Testing - Operational Acceptance Testing:
- validates if the system meets operational requirements. 
- Tests to include:
    Testing backup/restore
    High availability & Disaster recovery
    Testing maintenance tasks

Acceptance Testing - Regulatory Testing:
- Done to ensure the legal and compliance requirements are met.
- Validations include Sensitive data masked, Secure transactions, User consent captured, Logs maintained properly

                                    What is Shift Left Testing?
                                    ---------------------------

Traditionally:
    Dev builds -> QA tests later

Shift-left:
    Testing starts alongside or even before development

Testing on feature branch env is not shift left testing since its done after the code is written

Ideal shift left testing means
- Validating requirements before coding
- Writing test scenarios during feature grooming itself

How it looks in practice
- Writing test scenarios during requirement phase
- Reviewing user stories and acceptance criteria early
- Adding unit tests and API tests during development

In my current organisation, we have shift left practice, not that strict, 
we write and approve test design, 

-----------------------------------------------------------------------------------------------------------------
                                    
                                    RETESTING vs REGRESSION TESTING
                                    -------------------------------

Retesting:

- Done to verify a specific bug fix
- Focuses only on failed test cases
- Narrow scope, targeted

Regression Testing:

- Done to ensure new changes didn't break existing functionality
- Covers impacted + critical areas of the application
- Broad scope, suite-based
- Often automated and run in CI/CD, not done after every fix

What is QE (Quality Engineering)?

Quality Engineering is a modern quality approach that applies continuous
quality practices throughout the Software Development Lifecycle (SDLC).

QE is:
- Automation-driven
- Shift-left focused
- Closely integrated with Agile and DevOps
- Focused on building quality into the product from the beginning

Key Focus Areas:
- Test automation
- CI/CD quality gates
- Performance testing
- Security testing
- Observability
- Continuous testing

Example:
In a DevOps pipeline, QE ensures:
- Automated API/UI tests run on every build
- Security scans are automated
- Performance checks run continuously
- Defects are caught early in development

-----------------------------------------------------------------------------------------------------------

                                        Difference Between QE vs QA vs QC
                                        ---------------------------------

| QE (Quality Engineering)                            | QA (Quality Assurance)                         | QC (Quality Control)                          |
|-----------------------------------------------------|------------------------------------------------|-----------------------------------------------|
| Engineering-driven quality approach                 | Process-oriented approach                      | Product-oriented approach                     |
| Integrated with Agile, DevOps, CI/CD                | Ensures standards/processes are followed       | Validates final product quality               |
| Strong focus on automation and shift-left testing   | Mostly process and methodology focused         | Mostly execution/testing focused              |
| Continuous testing and monitoring                   | Verification activity                          | Validation activity                           |
| Proactive + Preventive + Continuous                 | Preventive/Proactive                           | Reactive                                      |
| Example: Automated quality pipelines                | Example: Defining QA standards/processes       | Example: Manual/automation test execution     |

One-line Difference:
- QA ensures quality processes.
- QC checks product quality.
- QE engineers quality throughout the entire SDLC.

                        Why does the cost of fixing defects increase in later phases of SDLC?
                        ---------------------------------------------------------------------

Because defects found late require more rework and  retesting.

Earlier phases:
- Defect is isolated
- Easier and cheaper to fix

Later phases:
- Code is already integrated/deployed
- Multiple modules may be affected and wider impact analysis is required
- Requires retesting/regression testing
- May impact customers/business

There is also- Deployment overhead
- Production support effort
- Reputation and revenue loss
