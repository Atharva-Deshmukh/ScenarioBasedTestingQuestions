Suppose you find a bug in production, how would you make sure the same bug is not introduced again?
- First, I’d do a quick root cause analysis to understand why it escaped—whether 
  it was due to missing test coverage, an edge case, or a gap in our automation or review process.

- Next, I’d create a automated test for the issue at the UI or API level.
  This test should fail first before the fix is deployed and pass after, 
  so we permanently capture the bug in our regression suite.

- Finally, I’d share the learnings with the team and update our test strategy and test suite

What has been your greatest challenge while doing regression testing?   

- Maintaining the regression suite as the application scaled.
- as more features were added, our regression suite became large, slow, and flaky. 
  Test execution time increased significantly, and failures were often due to instability rather than 
  real defects. This started impacting release timelines and reduced confidence in automation.
- I analysed the flaky tests and fixed them wherever possible, like locator update issue, testdata deletion
- I then prioritised the tests according to the service packs using tags, we modified regression tags
  Not everything needs to be run everytime

What are the drawbacks of the agile methodology / Implementation you faced?

- Sprints are usually very deadline constrained
- Documentation is not the priority
- while it promotes speed and flexibility, it can sometimes lead to insufficient clarity in requirements, 
  especially at the start of a sprint.
- In a few cases, user stories were not detailed enough, which caused misalignment between development 
  and testing. This led to rework, late defect discovery, and sometimes scope changes mid-sprint.
- Another challenge is compressed testing timelines. Since development and testing happen in the same sprint, 
  testing often gets pushed toward the end, leaving limited time for proper regression and edge case validation.
- In my current organisation, we earlier got the feature to test after it was fully coded and deployed to INT, 
  but this led to rework and delay when defects were found, so we started to deploy on feature environment first

What is your approach when you have a high priority release to be delivered in a very short time?

- my approach is to focus on risk-based testing and fast feedback, rather than trying to test everything.
- First, I quickly understand the scope and impact of the release — which areas of the system are affected, 
  and what the business-critical flows are.

  Based on that, I prioritize testing:
  - Critical user journeys (e.g., login, payments, core workflows)
  - High-risk areas (recent changes, complex logic, integrations)
  - I will do exploratory tests on impacted areas

- Then I rely heavily on automation:
  Run a smoke suite or critical regression suite in CI/CD to get quick confidence
  Ensure all existing stable automated tests are passing

Explain what will be your reaction if the project you have been working on got sudden change in its deadline?

- my first reaction is to stay calm and reassess the situation, rather than trying to rush testing blindly.
- Then I shift to a risk-based testing approach:

    Prioritize business-critical workflows
    Focus on areas impacted by recent changes
    Identify high-risk integrations or complex logic

    Next, I leverage automation for fast feedback:

    Run smoke or critical regression suites through CI/CD
    Ensure existing stable tests are passing

    For manual testing, I focus on:

    Exploratory testing in high-risk areas
    Edge cases that automation may not cover

    At the same time, I communicate clearly with stakeholders:

    Highlight what will be tested vs what may be partially tested - Partial release possibility
    Call out any risks or potential gaps early

How do you select regression test cases or form the regression test suite?

I keep it a risk-based approach.
- Identify modules affected by the recent changes.
- Prioritize critical business flows
- Include previously failed/bug-prone areas
- Prefer tests already automated, reliable, and fast (CI-friendly).
- Continuously optimize suite by Removing redundant tests, tag by priority (P0/P1/P2), and keep execution time efficient.

How do we decide if the test case is not ideal for automation?

- Highly unstable or frequently changing → high maintenance cost.
- One-time or rarely executed tests don’t justify automation effort.
- Visual look & feel, UX, or exploratory scenarios need human judgment.
- Third-party limitations CAPTCHA, OTP, external systems not controllable.

What is the most interesting bug that you have found recently?

One interesting bug I found was during exploratory testing in a rarely used part of the application

In our app flow: Sites → Entities → Components → Impacted Objects tab
This tab had a search bar to filter impacted objects by name

Issue:
  The search worked fine for normal text, but failed for names containing special characters like underscore (_)
  Since this tab was rarely used and a corner case, the issue was missed earlier

What I did:
  Reproduced and validated with different inputs
  Identified that the issue was due to a common controller-level filtering logic
  Extended testing across the application

Impact:
  Found that the same issue existed in multiple search fields across the app (especially with _ and &)
  Helped fix it at a centralized level, preventing multiple defects

As a tester, how do you differentiate a dummy login page which is exactly designed like your working project login page?

- Check API/network calls
  Real login will hit actual auth APIs and return tokens/session IDs; a dummy page won’t.
- Session/cookie handling
  Real app creates valid session cookies/JWT; dummy pages usually don’t.
- Logs & DB validation (if accessible)
  Confirm whether user authentication is recorded in backend systems.

How will you tell if enough test cases have been created to test your product?

- Go through all the acceptance criteria and check if all the requirements are mapped to atleast one testcase.
- For each test scenario, there should be positive, negative and boundary value test cases.
- Analyse the defects trend, fewer defects found over time -> indicates good coverage
- Regression suite completeness -> Core functionalities are part of regression and automated where possible.
- Stakeholder confidence -> Product, dev, and QA are aligned that major scenarios are validated.
